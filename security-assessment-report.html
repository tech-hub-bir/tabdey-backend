High1. Production secrets committed to git history

OWASP Category
A02 Cryptographic Failures
Files
.env files across root, admin, driver, food, mart, merchant, order_management, ride_management, sms-gateway-backend, wallet_payment, Map_and_location_services
Description
Multiple .env files were tracked in git and later removed from the working tree via commits like "Stop tracking env files" — removal does not purge git history, so contents remain recoverable via git show <old-commit>:<path>. Confirmed exposed: MySQL root password, MongoDB Atlas connection string, JWT/access/refresh secrets, Gmail app password, Redis password (incl. production), SMTP password, SMS master key, SMPP password.
Exploit Scenario
Anyone with repo read access runs git log --all --diff-filter=A -- wallet_payment/.env then git show <commit>:wallet_payment/.env to recover the production DB root password and connect directly, or forge valid JWTs for every service using the shared secret.
Recommendation
Treat all these credentials as compromised — rotate immediately. Purge from git history (e.g. git filter-repo) and add pre-commit secret scanning.

High2. BFS bank private signing key committed to git

OWASP Category
A02 Cryptographic Failures
File
bfs-wallet-backend/keys/grab.newedge.bt.key
Description
The PEM private key used by signChecksum() to sign every outbound transaction to the bank's BFS payment switch is checked into the repo and not gitignored.
Exploit Scenario
Anyone with repo access obtains the actual merchant signing key and can forge correctly-signed BFS payment requests, impersonating the merchant to the bank outside the application entirely.
Recommendation
Rotate the key/certificate pair with the bank immediately, remove from git history, load from a secrets manager, and gitignore keys/.

High3. BFS payment-response signature verification disabled

OWASP Category
A08 Software/Data Integrity Failures
Files
bfs-wallet-backend/src/utils/bfsChecksum.js:35-44, bfs-wallet-backend/src/services/bfsClient.js:155-164
Description
verifyChecksum is fully commented out. Every BFS response is trusted unconditionally — wallet credit is triggered purely on bfs_responseCode/bfs_debitAuthCode === "00" with no cryptographic verification the response came from BFS unmodified.
Exploit Scenario
A man-in-the-middle on the path to the BFS API can return a forged success code for a transaction the bank never approved, causing the platform to credit a wallet for a payment that never happened.
Recommendation
Re-enable verifyChecksum on every BFS response before trusting the result code; fail closed on missing/invalid checksum; enforce TLS to all BFS endpoints.

Overall Security Posture Summary

This backend exhibits a systemic, cross-service breakdown of authentication and authorization rather than isolated bugs: in multiple services (wallet_payment, grablike-backend, expo_push_notification, parts of admin/driver/merchant/food/mart, and the location Socket.IO layer), authentication middleware either does not exist, exists but is never wired into routes, or is explicitly disabled in source. This pattern directly enables critical financial abuse — unauthenticated wallet transfers, settlement adjustments, withdrawal approvals, and tip debits — and large-scale PII disclosure through unauthenticated lookup endpoints. Compounding this, the payment-integrity layer for the bank gateway (BFS) has its response-checksum verification commented out, meaning payment confirmations are currently trusted without cryptographic proof of authenticity, and the bank's private signing key, along with multiple generations of production database/JWT/SMTP/Redis credentials, are recoverable from git history even though current .gitignore rules hide them from a fresh checkout. Positively, the engineering team has clearly invested in some good patterns — bcrypt at an adequate cost factor, parameterized SQL/Prisma queries throughout (no SQL injection was found anywhere in the codebase), a centralized redacting logger and error handler, and amount validation guarding most wallet debit/credit paths against negative or non-numeric values — but these controls are undermined by the access-control and secret-management gaps above. Given the severity and breadth of the access-control findings on money-movement endpoints and the compromised payment-integrity check, this backend should not be considered production-ready for a payment-handling clearance until the High-severity findings (secret rotation, route-level auth enforcement across all services, and re-enabling BFS checksum verification) are remediated and independently re-verified.
